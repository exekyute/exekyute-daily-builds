# Spec: property tax burden index

## Purpose

Measure how much harder each Nova Scotia municipality taxes commercial
property than residential property, rank every municipality by that gap
within each fiscal year, flag the widest gaps in the latest year, and track
each municipality's gap over time.

## Inputs

One pinned snapshot, committed at
`data/raw/ns_property-tax-rates_2026-07-06.csv` (1,112 data rows). Columns as
published: `area`, `area_type`, `year` (fiscal label like `2025/2026`),
`residential`, `commercial`. Both rates are general tax rates in dollars per
$100 of assessed value. Provenance in SOURCE.md.

## Cleaning rules (sql/02_transform.sql)

1. `area` and `area_type` are trimmed of surrounding whitespace.
2. `year_start` = the first four characters of the trimmed fiscal label, cast
   to INTEGER (`2025/2026` gives 2025). The label itself is kept as
   `year_label`.
3. `residential` and `commercial` are cast to DOUBLE with `try_cast`. A row
   where the year or either rate fails to parse is dropped rather than
   guessed at.
4. One row per `(area, area_type, year_start)`. The name alone is NOT a key:
   six names (Antigonish, Digby, Lunenburg, Pictou, Shelburne, Yarmouth) are
   both a Town and a Rural Municipality, and those are different
   municipalities with different rates. If the source ever carries a true
   duplicate on the full key, the row with the lowest residential rate wins,
   then the lowest commercial rate; `row_number()` makes the choice
   deterministic.

On the 2026-07-06 snapshot: rule 3 drops exactly one row (COUNTY SUBURBAN,
2013/2014, both rates null in the source) and rule 4 drops nothing, so 1,111
of 1,112 rows survive. The six paired names account for the 102 rows that a
name-only key would have wrongly collapsed.

## Analysis, step by step (sql/03_analysis.sql)

1. **Spread.** `spread = round(commercial_rate - residential_rate, 4)`. This
   is the burden index: the extra dollars per $100 of assessed value that a
   commercial property pays over a residential one.
2. **Rank within year.** `rank_in_year = RANK() OVER (PARTITION BY year_start
   ORDER BY spread DESC)`. Widest spread is rank 1; equal spreads share a
   rank and the next rank is skipped.
3. **Outlier rule (exact).** Let `latest_year = MAX(year_start)` over the
   cleaned data and `p90 = quantile_cont(spread, 0.90)` computed over the
   latest-year rows only (linear interpolation). Then
   `is_outlier = (year_start = latest_year AND spread >= p90)`. The flag is
   false for every earlier year by construction.
4. **Year-over-year movement.** `yoy_spread_change = round(spread -
   LAG(spread) OVER (PARTITION BY area, area_type ORDER BY year_start), 4)`.
   NULL in a municipality's first observed year. If a municipality skips a
   year, LAG compares against its previous observed year, not the missing
   calendar year.

## Dashboard re-derivation

`dashboard/data.js` is regenerated by `run.py` from the mart on every run and
holds the mart rows as a `const DATA` literal, nothing more. The page then
recomputes its figures in JavaScript from `DATA`:

- latest year = max of `year_start`
- headline = the latest-year row with the widest `spread` (area name then
  area type break a tie, same as the SQL export order)
- median = sort latest-year spreads ascending, average the middle two when
  the count is even, round to 4 decimals (the same rule as SQL `median()`)
- outlier count = latest-year rows with `is_outlier` true

The exact values the page must show, equal to the golden output:

- Widest spread: **3.9000**, CLARK'S HARBOUR (Town), fiscal 2025/2026
  (commercial 5.58, residential 1.68), rank 1 of 64 municipalities
- Provincial median spread: **1.9760** across 64 municipalities
- Outliers flagged: **9** (the 90th-percentile threshold works out to 3.1901:
  Clark's Harbour, Mulgrave, and the seven Cape Breton Regional communities
  tied at 3.1901)

Two checks confirmed these values: the derivation functions in
`dashboard/dashboard.js` ran headlessly against the generated `data.js` with
assertions on every figure above, and the rendered page showed the same
numbers in a browser.

## Outputs

- `out/tax_burden_index.csv`: the full index, one row per municipality per
  year, ordered by `year_start`, `rank_in_year`, `area`, `area_type`.
  Golden-tested against `expected/tax_burden_index.csv`.
- `bi/exports/mart_tax_burden.csv`: the same rows, written for Power BI and
  the dashboard.
- `dashboard/data.js`: the mart as a JS literal, regenerated each run.

Column definitions in data_dictionary.md.

## Edge cases

- **Two municipalities sharing a name:** six names are both a Town and a
  Rural Municipality. Every partition, join, and dashboard lookup keys on
  `(area, area_type)`, and the dashboard labels them "NAME (Type)".
- **Duplicate `(area, area_type, year)` rows in the source:** resolved by
  rule 4, deterministically (none in the pinned snapshot).
- **Unparseable rate or year:** dropped by rule 3; never silently zeroed.
- **Rank ties:** possible in principle; RANK() shares the rank, and the
  export's `ORDER BY year_start, rank_in_year, area` keeps row order stable
  regardless.
- **A municipality's first year:** `yoy_spread_change` is NULL (empty in the
  CSV, `null` in data.js).
- **Gap years:** LAG spans the gap; the change is against the previous
  observed year.
- **Even row count at the median or the 90th percentile:** both
  `quantile_cont` and the JS re-derivation interpolate linearly, so they
  agree by construction.
- **Floating point:** every derived number is rounded to 4 decimals in SQL
  before export, so the golden diff compares clean strings.

## Determinism

Same snapshot in, same bytes out. The snapshot is pinned and committed, every
result query ends in ORDER BY, the dedup rule is a total order, and nothing
in the pipeline draws on time, randomness, or environment. `python run.py`
re-derives everything and must print PASS against
`expected/tax_burden_index.csv`; that golden file was produced by the first
verified run and confirmed stable on a re-run.
